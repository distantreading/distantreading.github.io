Encoding for ELTeC : an update

ELTeC Encoding Requirements (a reminder!)

  • support computational approaches to literary text analysis (authorship attribution, topic modelling, stylistic analysis ...)
  • enrich corpora with metadata and impose only a minimal structure
  • editorial issues of lesser interest
  • markup should offer the encoder very little choice, and the software developer very few surprises
  • aim to facilitate uniform and consistent access across multiple corpora

Traditional TEI, by contrast, rejoices in variety, which makes comparative work harder

ELTeC encoding scheme/s

  • level 0: minimal encoding scheme for texts produced manually or by OCR from print originals
  • level 1 : somewhat richer format derivable automatically from texts encoded in other formats (Word, HTML TEI ...)
  • level 2 : lingistically annotated and segmented

and a tightly constrained Header common to each level

Where are we now?

  • In June 2019, with several hundred texts already done, we reviewed some proposed changes to the schema, almost all of them in the header
  • Typical examples: constrain type values on <bibl> and on <div>; add constraints on presence of <note>s
  • A new XSLT stylesheet converts existing valid files to files which are valid with the new schema
  • We have tested this on all currently available files
  • It is planned to convert all files to use the new schema before the first Zenodo release

ELTeC Encoding level 2

  • We have developed but not much tested a proposal for embedding linguistic annotation within existing level0 or level1 files
  • This could be used to define training sets, or to express the results of automatic word level annotation (NER, pos, etc.)

We circulated details to WG2 in June 2019, but there has been little or no feedback, and none of the experimental work going on currently in WG2 seems to produce texts in a TEI format.
